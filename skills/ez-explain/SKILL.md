# Ez-Explain

IRON LAW: 输出文章中不得出现任何与信息源相同的完整句子，不得出现任何与信息源相同的代码示例。每个知识点必须用全新的表达方式呈现，每个示例必须重新构造——相同的知识，不同的话；相同的原理，不同的例子。

## Workflow

```
Rewrite Progress:

- [ ] Step 1: 收集信息源 ⛔ BLOCKING
  - [ ] 1.1 确认所有输入源（文件路径/URL/粘贴内容）
  - [ ] 1.2 读取并解析每篇源文档
  - [ ] 1.3 统计：N 篇来源 → 1 篇输出
- [ ] Step 2: 验证信息源质量 ⚠️ REQUIRED
  - [ ] 2.1 标注每篇来源的发布/更新时间
  - [ ] 2.2 识别可能过时的内容（版本号、API 变更、已废弃特性）
  - [ ] 2.3 对过时内容联网搜索最新信息
  - [ ] 2.4 标注事实性存疑的论述
  - [ ] 2.5 生成验证报告
- [ ] Step 3: 提取知识骨架 ⚠️ REQUIRED
  - [ ] 3.1 从所有来源中提取核心知识点
  - [ ] 3.2 建立知识点之间的逻辑链（因果/递进/对比）
  - [ ] 3.3 识别多篇来源的重叠与互补部分
  - [ ] 3.4 融合为统一的知识图谱
- [ ] Step 4: 设计文章结构 ⚠️ REQUIRES CONFIRMATION
  - [ ] 4.1 确定文章主题和切入角度
  - [ ] 4.2 设计章节大纲（展示给用户）
  - [ ] 4.3 等待用户确认大纲
- [ ] Step 5: 写作 ⚠️ CRITICAL
  - [ ] 5.1 加载 references/style-spec.md
  - [ ] 5.2 按大纲逐节写作
  - [ ] 5.3 确保每个知识点的表达与来源完全不同
  - [ ] 5.4 补充来源中缺失但逻辑上必要的衔接内容
  - [ ] 5.5 为复杂知识点生成 Mermaid 图辅助理解
- [ ] Step 6: 自检 ⚠️ REQUIRED
  - [ ] 6.1 逐段对比来源，确认无相同句子
  - [ ] 6.2 检查知识点覆盖率
  - [ ] 6.3 检查逻辑连贯性
  - [ ] 6.4 检查过时信息是否已更新
  - [ ] 6.5 AI 腔检查：排查模板化措辞和套话
- [ ] Step 7: 交付
  - [ ] 输出最终文章
  - [ ] 附知识来源映射表（可选）
```

## Parameters

- `$ARGUMENTS` — 信息源路径或 URL（可多个，空格分隔）
- `--output <path>` — 输出文件路径（默认：打印到终端）
- `--style <name>` — 写作风格（默认：tech-conversational）
- `--outline-only` — 仅生成大纲，不写正文
- `--no-verify` — 跳过信息源验证（如已确认来源可靠）

## Step 1: 收集信息源

接受以下输入形式：
- 本地文件路径（.md / .txt / .html）
- 公网 URL（自动抓取内容）
- 对话中粘贴的文本块

### URL 读取策略

当输入包含 URL 时，按以下顺序尝试读取：

1. **WebFetch**（优先）：适用于大部分公开博客、文档站、GitHub。直接抓取并转为文本。
2. **Playwright 浏览器**（兜底）：当 WebFetch 失败或返回空内容时使用——适用于需要 JS 渲染的页面（SPA、飞书/Notion/语雀等在线文档）。
   - 导航到 URL，等待页面加载
   - 使用 `browser_snapshot` 获取可访问性树或使用 `browser_evaluate` 提取正文内容
   - 关闭页面释放资源

判断依据：
- 静态页面（博客、GitHub、MDN）→ WebFetch
- 在线文档平台（飞书、Notion、语雀、Google Docs）→ Playwright
- 不确定 → 先 WebFetch，失败后 fallback 到 Playwright

如果 URL 需要登录才能访问，告知用户无法读取，请求其将内容粘贴到对话中。

多源输入规则：
- 无论输入几篇来源，输出始终为**一篇**融合文章
- 多篇来源中的知识点取并集，重复内容去重
- 矛盾内容标注冲突，联网验证后取最新/最准确的版本

## Step 2: 验证信息源质量

对每篇来源执行以下检查：

### 时效性检查

问：这篇内容的核心信息是否仍然准确？

- 提到的 API/SDK 版本是否为当前版本？
- 引用的数据/统计是否有更新？
- 描述的工具/框架行为是否已变更？

如发现过时内容：
1. 使用 WebSearch 搜索该知识点的最新状态
2. 对比来源描述与最新信息
3. 在知识骨架中标注需要更新的部分

### 准确性检查

问：这篇内容的事实论述是否经得起验证？

- 技术原理描述是否准确？
- 代码示例是否能正确运行？
- 因果关系推导是否合理？

对存疑内容联网交叉验证。

### 验证报告格式

```
信息源验证报告
==============
来源 1: <标题/路径>
  时效性: ✅ 当前 | ⚠️ 部分过时 | ❌ 严重过时
  过时项: [列出具体过时内容及最新信息]
  准确性: ✅ 可靠 | ⚠️ 存疑（列出）

来源 2: ...

综合判断: 可直接使用 | 需更新后使用 | 建议补充更多来源
```

⚠️ 验证报告展示给用户确认后再继续。

## Step 3: 提取知识骨架

从所有来源中提取：

1. **核心概念**：每个来源教了什么
2. **逻辑链**：概念之间的关系（A 导致 B、A 是 B 的前提、A 和 B 对比）
3. **关键数据**：数字、代码、对比表格
4. **实践要点**：注意事项、常见错误、最佳实践

多源融合规则：
- 同一知识点在多篇来源中出现 → 取最完整、最准确的版本
- 不同来源提供互补视角 → 在文章中融合呈现
- 来源之间有矛盾 → 验证后取正确的，注明另一种观点存在

## Step 4: 设计文章结构 ⚠️ REQUIRES CONFIRMATION

基于知识骨架设计文章大纲。

设计原则：
- 从读者能感知的场景切入（不从定义开始）
- 渐进式展开：简单 → 复杂，表象 → 本质
- 每个章节有明确的"教学目标"
- 章节之间有自然过渡

大纲格式：
```
文章标题: <标题>
切入角度: <一句话描述从哪个场景引入>

章节大纲:
  1. <章节标题> — <这节教什么>
  2. <章节标题> — <这节教什么>
  ...

预计篇幅: ~N 字
```

⚠️ 展示大纲，等待用户确认或调整后再写作。

## Step 5: 写作

加载 `references/style-spec.md`，按照风格规范写作。

### 原创表达要求

对每个知识点，执行以下转换：

1. **换角度**：来源从 A 角度解释？你从 B 角度切入
2. **换比喻**：来源用比喻 X？你用完全不同的比喻 Y
3. **换结构**：来源先说原理再说现象？你先说现象再拆原理
4. **换粒度**：来源笼统概括？你用具体例子展开；来源举了例子？你提炼出更一般的规律

禁止：
- 直接使用来源的句子（即使改了个别词）
- 照搬来源的段落结构
- 复制来源的代码注释（代码逻辑可参考，注释必须重写）
- 复制来源的代码示例（同一个原理必须用不同的场景/变量/数据演示）

### 示例重构要求

所有示例（代码、JSON、表格、对比）必须重新构造，不可复用来源的示例：

1. **换场景**：来源用"天气查询"演示？你用"订单查询"或"文件搜索"
2. **换数据**：来源用 `{city: "北京"}`？你用完全不同的字段和值
3. **换变量名**：来源叫 `retryDelay`？你用 `backoffInterval` 或完全不同的命名
4. **换结构**：来源用 3 步流程演示？你拆成 2 步或展开为 4 步

代码示例检查标准：遮住来源代码，仅凭知识点描述，独立写出演示代码。如果写出来和来源几乎一样——说明场景选得太近，换一个。

### 写作节奏

- 开场 2-3 段建立场景和问题
- 每个知识点 3-5 段（背景 → 机制 → 注意事项）
- 知识点之间 1 段过渡
- 结尾不做空洞总结，以"下一步"或"延伸思考"收束

### 段落衔接（承上启下）

每个大节的开头必须完成两件事：总结前文落点 + 引出本节要解决的新问题。不能假设读者记得上一节的结论。

具体要求：
- **节与节之间**：上一节的最后 1-2 句应预告"还有什么没解决"，下一节的第一句应接住这个未解决的问题。如果上一节末尾没有自然的预告位，在下一节开头用一句话点明"前面讲了 X，但 X 还不够，因为 Y"。
- **子标题之间**：同一节内的子标题切换时，用一句话说明新子标题和前一个子标题的关系（并列、递进、转折）。不能让读者自己猜"这两个小节是什么关系"。
- **禁止孤立起笔**：任何一个小节的第一句话，不能只讲自己的内容而完全不提前文。读者冷启动读到任何一节，都应该能从开头 1-2 句判断出"这节在整篇文章里的位置"。

检验方法：遮住上一节，只看当前节的第一段。如果无法判断它在讲文章的第几层问题、前面已经解决了什么，说明衔接缺失。

### 开场真实性

开场示例必须来自读者真实会遇到的场景，不要编造假设性场景。优先使用：
- 用户提供的真实项目代码/配置
- 真实的 API 响应格式（如 Anthropic 的 `tool_use` 格式需要带 `type`、`id`、`name`、`input`）
- 真实的工具名和参数结构

禁止：
- 使用"假如你在 ChatGPT 里输入..."这种用户实际看不到底层 JSON 的场景
- 编造不存在的 API 格式或参数名
- 使用简化到失真的 JSON 示例（缺少必要字段）

### 示例一致性

全文的示例必须前后呼应：
- 开头引入的示例（工具名、字段名、场景）是全文的锚点
- 后续章节引用"开头那个例子"时，引用的字段必须在开头 JSON 中实际存在
- 不要在中间章节突然引入从未出现过的字段名或工具名（如开头用 `askQuestion` 工具，后面突然冒出 `departure` 字段）
- 写完后全文搜索所有出现的字段名/工具名，确认每一个都能在上文中找到出处

### 关键名词标记

技术术语和专有名词用 backtick 标记：`Function Calling`、`JSON Schema`、`Prompt Cache`、`KV Cache` 等。判断标准：如果这个词在代码或配置中会原样出现，就加 backtick。中文描述性词汇不加。

### 参考资料与数据引用

文中引用具体数据（百分比、性能数字、对比结论）时，必须在引用处标注脚注编号（如 [1]），并在文末提供带完整 URL 的参考资料列表。

参考资料格式：
```
**参考资料**

1. [标题](URL) — 一句话说明引用了什么
2. ...
```

需要标注脚注的内容：
- 具体的百分比或数字（"93% 格式合规率"）
- 产品/团队的公开声明（"Manus 团队的经验总结"）
- 安全漏洞和攻击案例
- 不是常识的技术断言

### Mermaid 图生成

对复杂度高或涉及多步交互的知识点，生成 Mermaid 图辅助理解。

判断标准——以下情况必须配图：
- 涉及 3 个以上组件/角色的交互流程 → `sequenceDiagram`
- 涉及条件分支或决策逻辑 → `flowchart TD`
- 涉及状态变迁 → `stateDiagram-v2`
- 涉及层次结构或系统组成 → `flowchart LR` 或 `block-beta`

图表规范：
- 节点标签用中文
- 边标签简短（2-6 字）
- 控制在 8-15 个节点（太少没信息量，太多看不清）
- 时序图（sequenceDiagram）的消息标签必须带编号：`1.1`、`1.2`、`2.1`...，方便正文引用
- 图表前后各一句话：前面说"这是什么"，后面说"关键要看哪里"
- 不要为简单的线性流程配图（文字描述足够）
- 多步骤流程优先用 subgraph 分组（如"输入检查"/"外部控制"/"执行"），而不是画一条长链。分组让读者一眼看出阶段划分
- 流程图优先横向（LR）而非纵向（TD），横向更节省页面空间且符合阅读流向

### AI 腔检查

写完后扫描全文，排查以下模板化表达并替换：

黑名单词汇（发现即改）：
- "真相"、"揭秘"、"底牌"、"亮底牌"
- "生产级"、"工业级"、"企业级"（除非在讨论具体的生产环境部署）
- "名字撞车"、"填进去的肉"、"骨架"等过度比喻
- "优雅地"、"不优雅"
- "银弹"、"没有银弹"
- "开箱即用"、"一站式"
- "赋能"、"沉淀"、"抓手"
- "痛点"、"打法"、"闭环"
- "深入浅出"、"由浅入深"
- "接下来让我们..."、"让我们一起看看..."

黑名单标点/符号（发现即改）：
- `「」`（直角引号）：全部替换为 `""`（双引号）或去掉引号。`「」` 是 AI 生成中文的标志性痕迹，人类写作极少使用。
- `——`（破折号）过度使用：全文不超过 3 处。大量 `——` 是 AI 写作的典型特征。替换为句号断开、逗号连接、或冒号引出。每处 `——` 都要判断：这里用逗号或句号是否更自然？如果是，就换掉。

检查方法：
1. 用 grep 搜索 `「` `」` `——`，确认 `「」` 为零、`——` 不超过 3 处
2. 逐段朗读——如果某句话像是"AI 写的模板"，换掉
3. 检验标准：这句话是不是只有在这篇文章的这个具体语境下才说得出来？如果换个话题也能说——就是套话，删掉或重写

## Step 6: 自检

### 原创性检查

逐段执行：
1. 在来源中搜索是否有相同或高度相似的句子
2. 如发现相似度过高的表达 → 立即重写该句
3. 检查标准：同一个知识点的表述，遮住来源后能否独立写出完全不同的句子

### 示例原创性检查

逐个检查所有代码块、JSON 示例、表格：
1. 对比来源中的示例——场景、变量名、数据值是否重复
2. 如发现场景相同（哪怕变量名不同）→ 换一个完全不同的场景重写
3. 检查标准：读者同时看到来源和输出，不会觉得"这是同一个例子改了改"

### 知识完整性检查

- 来源中的每个核心知识点是否都在文章中有对应
- 来源中的关键数据是否被正确引用（数字不能改）
- 过时内容是否已替换为最新信息

### 逻辑检查

- 章节之间是否有自然过渡（1-2 句点明上下文关系即可，不要模板化的"接下来让我们看看"）
- 过渡句要说清"前面讲了什么，后面为什么要讲这个"，而不是刻意铺垫
- 因果关系是否清晰
- 读者是否能不回看前文地理解当前内容

### 示例一致性检查

- 全文搜索所有出现的字段名、工具名、变量名
- 确认每一个都能在上文中找到首次引入的位置
- 如果某个名称只在某一段出现且从未被介绍过，要么补充引入说明，要么换成已有的名称
- 重点检查：引用"开头的例子"时，引用的字段是否真的在开头 JSON 中存在

### 符号与格式检查

用 grep 执行以下检查：
- `「」` 残留数：必须为 0
- `——` 出现数：不超过 3 处
- AI 腔黑名单词汇：必须为 0
- 技术术语是否加了 backtick
- 数据断言是否标了脚注编号
- 文末参考资料是否有完整 URL

## Anti-Patterns

- **句子级搬运**：把来源的句子换几个同义词就当原创——必须整句重新构思
- **结构克隆**：保留来源的段落顺序和章节结构，只换措辞——结构本身也要重新设计
- **示例换皮**：来源用"天气查询"你用"天气预报"，场景本质相同——必须换到完全不同的领域
- **代码改名**：把来源代码的变量名改了就当新代码——逻辑、场景、数据都要重新设计
- **遗漏核心知识**：为了追求"不同"而丢掉来源的关键信息——知识必须完整保留
- **无中生有**：来源没有的观点，不要编造后当作来源的内容写入
- **忽略验证**：不检查时效性就直接重写——过时信息传播比不写更有害
- **学术腔**：写出"综上所述"、"由此可见"式的论文语言——保持对话感
- **空洞开场**："随着 AI 技术的快速发展..."——从具体场景切入
- **AI 套话堆砌**：用"真相"、"揭秘"、"生产级"、"优雅地"等模板词填充——用具体描述替代
- **强行配图**：简单的线性流程也画 Mermaid 图——只有多角色交互或条件分支才需要图

## Pre-Delivery Checklist

- [ ] 无任何与来源相同的完整句子
